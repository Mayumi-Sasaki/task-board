## Git 運用ルール

### 基本方針

**コードを変更するたびに、必ずコミットして GitHub にプッシュすること。**

変更を溜め込まず、作業単位ごとに即座にリモートへ反映する。

### コミットの粒度

- 1つの論理的な変更（機能追加・バグ修正・リファクタなど）= 1コミット
- 複数の変更をまとめてコミットしない
- 作業途中の壊れた状態でコミットしない

### コミットメッセージ

```
<type>: <変更内容の要約（日本語可）>
```

| type | 用途 |
|------|------|
| feat | 新機能追加 |
| fix | バグ修正 |
| refactor | 動作を変えないコード改善 |
| style | フォーマット変更のみ |
| test | テストの追加・修正 |
| docs | ドキュメントのみの変更 |
| chore | ビルド・設定ファイルの変更 |

### プッシュの手順

コード変更後は必ず以下を実行する：

```bash
git add <変更ファイル>
git commit -m "<type>: <メッセージ>"
git push origin <ブランチ名>
```

### ブランチ運用

- `main` ブランチへの直接コミットは原則禁止
- 作業は `feature/<機能名>` または `fix/<バグ名>` ブランチで行う
- レビュー後に Pull Request 経由で `main` にマージする

### 禁止事項

- `git push --force` は原則禁止（明示的に許可された場合のみ）
- `--no-verify` でフックをスキップしない
- `.env` などの機密情報を含むファイルをコミットしない
